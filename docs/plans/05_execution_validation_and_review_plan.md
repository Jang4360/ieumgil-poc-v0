# 05. 실행 검증, 테스트 하네스, 리뷰 기록 계획

## 목적

보행 네트워크 생성 `2~6번`과 GraphHopper import `10~13번` 구현이 문서로만 끝나지 않도록, 실행 검증과 리뷰 기록 방식을 고정한다.

## 구현 범위

- 파서/세그먼트 생성 검증
- 저장 구조/안정 키 검증
- import loader 검증
- artifact 생성/로드 검증
- 리뷰 문서 작성 규칙 적용

## 비범위

- 공공데이터 속성 정확도 검증
- 대중교통 접근성 검증
- 프런트엔드 UI 검증

## 구현할 내용

### 1. 파서와 세그먼트 검증

- 동일 입력 PBF에 대해 반복 실행 시 동일한 `eligible_osm_way`, anchor, segment 결과가 나오는지 확인한다.
- `road_segment`가 `way` 전체가 아닌 anchor 구간이라는 점을 예시 케이스로 검증한다.

### 2. 저장 구조 검증

- `road_nodes`에 non-anchor node가 들어가지 않는지 검증한다.
- 안정 키 중복, 역방향 중복, 누락 필드를 검출하는 검증 규칙을 만든다.

### 3. import 검증

- loader가 안정 키 lookup으로 segment를 찾는지 검증한다.
- import 중 per-edge query가 없는지 설계 또는 로그 기준으로 확인한다.
- artifact 생성 후 로드 경로가 문서와 일치하는지 확인한다.

### 4. 리뷰 기록

- 각 구현 단위는 성공/실패 여부와 상관없이 `docs/reviews/`에 결과를 남긴다.
- 파일명은 `YYYY-MM-DD_<plan-id>_<success|failure>_review.md` 형식을 따른다.

## 완료 기준

- 각 작업 단위마다 검증 가능한 입력과 기대 결과가 정의되어 있다.
- 파서, 세그먼트 생성, 적재 구조, import loader, artifact 로드에 대한 최소 1개 이상 검증 방법이 명시되어 있다.
- 구현 결과가 성공이든 실패든 `docs/reviews/`에 기록되는 운영 규칙이 문서로 고정된다.

## 재시도 및 리뷰 작성

- 검증 체계가 완료 기준을 만족하지 못하면 원인 수정 후 최대 2회 재시도한다.
- 2회 재시도 후에도 해결 방안이 없으면 `docs/reviews/`에 실패 결과, 누락된 검증 포인트, 다음 액션을 `md` 파일로 남긴다.
- 성공한 경우에도 `docs/reviews/`에 성공 결과, 사용한 검증 입력, 남은 테스트 공백을 `md` 파일로 남긴다.
